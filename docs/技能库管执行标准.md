# 技能库管执行标准

更新日期：2026-05-06

本文档用于固定“技能库管”和“技能注册表”的长期执行要求。以后只要新增、整理、评价或发布 skill / prompt，默认按照本文档执行，不需要重复确认这些基础需求。

## 1. 核心原则

1. 技能和 Prompt 必须独立保存，不能继续堆叠在一个总文件里。
2. 每个技能都必须能被未来任务直接定位、读取和调用。
3. 注册表不是资料仓库，而是“索引、判断规则、学习路线和调用路由”。
4. 原始开源项目的代码、授权和下载入口以原仓库为准，本库只做整理和索引。
5. 不编造技能效果、不编造推荐来源、不把普通口碑包装成专家背书。
6. 新增信息里只要出现可复用 skill 或 prompt，就要沉淀为独立文件。

## 2. 目录标准

本地技能库根目录：

```text
E:\L-One知识库\L-One知识库\具体有用的技能库\codex-video-tools-下载1-2026-05-06
```

固定目录职责：

| 目录 | 职责 |
|---|---|
| `05-independent-skills/` | 每个技能一个独立文件夹，核心入口为 `SKILL.md` |
| `06-independent-prompts/` | 每个 Prompt 一个独立 Markdown 文件 |
| `07-skill-registry/` | 注册表、路由规则、工作流模板、可视化 HTML、执行标准 |

旧的原始资料、下载证据、混合清单和总堆叠文件不作为技能调用入口。已经整理过的旧资料应删除或降级为不可调用参考，避免未来误读。

## 3. 独立技能文件标准

每个技能必须独立成目录：

```text
05-independent-skills/<skill-id>/SKILL.md
```

`SKILL.md` 至少包含：

1. 技能名称和一句话用途。
2. 适合解决的问题。
3. 不适合或容易误用的场景。
4. 核心工具、依赖或原始项目。
5. 默认执行流程。
6. 质量检查标准。
7. 常见失败点和排错经验。
8. 可联动的其他技能。
9. 相关 Prompt。
10. 原始来源链接和授权提醒。

同一个 `SKILL.md` 不混写多个独立技能。技能之间可以互相引用，但不能互相吞并。

## 4. 独立 Prompt 文件标准

每个可复用 Prompt 必须独立保存：

```text
06-independent-prompts/<number>-<prompt-id>.md
```

Prompt 文件至少包含：

1. Prompt 名称。
2. 适用场景。
3. 完整可复制 Prompt。
4. 输入变量说明。
5. 预期输出格式。
6. 适合联动的技能。
7. 使用后的经验回写要求。

Prompt 不只放在网页卡片中。网页可以展示摘要，但真实可调用版本必须在独立文件中。

## 5. 技能注册表标准

技能注册表必须同时服务两个对象：

1. 给人看：快速理解每个技能的用途、状态、可信度和来源。
2. 给未来任务调度用：判断什么时候该调用哪个技能，哪些技能需要联动。

注册表至少维护：

1. 技能 ID。
2. 主分类和细分类。
3. 认可星级。
4. GitHub 星数。
5. 最近更新日期。
6. 当前状态。
7. 一句话理解。
8. 主效果。
9. 适合任务。
10. 避免误用。
11. 联动技能。
12. 相关 Prompt。
13. 质量门槛。
14. 原始来源链接。
15. 推荐和使用引用。

## 6. 网页展示标准

可视化网页文件：

```text
07-skill-registry/skill-registry.html
```

发布仓库入口：

```text
C:\Users\Administrator\Documents\New project 2\L-One-skills\index.html
```

线上地址：

```text
https://macabyavaha7-sys.github.io/L-One-skills/
```

网页结构标准：

1. 左侧必须有清晰分类菜单，并支持单页内子页面切换。
2. 首页展示仓库状态、总技能数量、活跃技能数量、Prompt 数量、分类比例和技能卡片。
3. 每个分类页面展示该分类下的技能和统计。
4. 技能以独立卡片展示，卡片顶部必须显示主分类大标题。
5. 点击技能卡片打开弹窗，展示完整详情。
6. 首页底部不再放大块“推荐工作流”和“Prompt 支援”。
7. Prompt 在技能弹窗中展示，同时另设左侧菜单 `Prompt / AI 技巧` 供单独查看。
8. 页面底部只保留轻说明：

```text
所有技能仅作索引与学习路线整理，实际授权与下载请以原始仓库为准。
```

## 7. 视觉设计标准

当前固定风格：

1. 灰色背景。
2. 深色左侧菜单。
3. 白色大圆角主面板。
4. 细线条、轻阴影、低噪音布局。
5. 橙色用于重点数字、星级和关键状态。
6. 重要数据要放大显示，例如技能数量、活跃技能、Prompt 数量。
7. 不显示无用指标，例如“平均认可”。
8. 顶部右侧分类比例图要有足够视觉占比。
9. 宽屏三列技能卡，中屏两列，窄屏单列。
10. 移动端必须自适应宽度，避免文字溢出和板块挤压。

## 8. 技能卡片标准

每张技能卡至少显示：

1. 主分类大标题。
2. 技能名称。
3. 一句话简介。
4. 主效果摘要。
5. GitHub 星数。
6. 最近更新日期。
7. 认可星级。
8. 图标。

技能卡片不是下载按钮。下载、授权和源码访问都通过弹窗中的原始 GitHub 链接完成。

## 9. 技能弹窗标准

每个技能弹窗必须展示：

1. 认可星级。
2. GitHub 星数。
3. 最近更新日期。
4. 状态。
5. 一句话理解：说明这个技能具体解决什么问题、适合什么场景。
6. 主效果。
7. 适合任务。
8. 避免误用。
9. 联动技能。
10. 相关 Prompt。
11. 质量门槛。
12. 原始来源 / GitHub 链接。
13. 推荐和使用引用。
14. 授权提醒。

弹窗中的授权提醒固定为：

```text
请以原始仓库的 README、LICENSE 和发布说明为准。本页面只做技能索引与学习路线整理。
```

## 10. GitHub 星数标准

1. 卡片显示该技能关联原始 GitHub 仓库的 star 总量。
2. 弹窗来源列表显示每个原始仓库的单独 star。
3. 发布前必须运行同步脚本抓取当前星数。
4. 如果 GitHub API 返回 403，可以通过公开 GitHub 页面解析 star 数。
5. 除非确实无法获取，否则不要长期显示“待同步”。
6. 如果无法获取，必须标注原因和最后尝试日期。

当前发布脚本：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\publish-skill-registry.ps1
```

执行目录：

```text
C:\Users\Administrator\Documents\New project 2\L-One-skills
```

## 11. 认可星级标准

认可星级表示“是否值得在真实任务中调用”，不是单纯的人气评分。

| 星级 | 含义 | 默认调度判断 |
|---|---|---|
| 1 星 | 线索级 | 只作为候选线索，不直接进入关键流程 |
| 2 星 | 可参考 | 可用于辅助判断，需要人工复核 |
| 3 星 | 可调用 | 能处理明确小任务，但需要质量检查 |
| 4 星 | 稳定可用 | 可作为常规工作流组件 |
| 5 星 | 核心技能 | 可作为复杂任务的主技能或关键节点 |

星级判断要综合：

1. 是否解决真实高频问题。
2. 是否有稳定工具或成熟项目支撑。
3. 是否容易被未来任务复用。
4. 是否能和其他技能组成工作流。
5. 是否有清晰边界和质量门槛。

## 12. 推荐引用标准

“大家如何推荐和使用”必须严格可验证：

1. 每个技能最多展示 Top 5 条引用。
2. 总字数控制在 500 字以内。
3. 优先级：官方文档、原作者说明、领域专家、工程团队文章、高质量社区讨论、课程或公开资料。
4. 可以引用 X / 社区大神推荐，但必须有可访问链接。
5. 不得编造专家、平台、评价或推荐理由。
6. 没有可信来源时写“待补充可信引用”，不要硬凑。

每条引用应说明：

1. 来源是谁。
2. 推荐或使用这个项目的理由。
3. 它说明了技能的哪种真实价值。
4. 原始链接。

## 13. 来源和授权标准

1. 本技能库不替代原始仓库。
2. 不在网页中重新打包别人源码供下载。
3. 不隐藏原作者来源。
4. 每个技能弹窗必须放原始 GitHub 链接。
5. 用户或访问者想下载技能时，应跳转到原始创作者主页。
6. 实际授权、安装、下载、版本和风险以原仓库 README、LICENSE、Release 为准。

## 14. 多技能工作流标准

复杂任务不能只看单个技能，要根据任务路径选择主技能和辅助技能。

默认判断顺序：

1. 先判断任务目标属于哪个主分类。
2. 选择一个主技能负责核心产出。
3. 根据输入来源、格式、发布方式、质量门槛选择辅助技能。
4. 为每个节点匹配相关 Prompt。
5. 最后用质量门槛检查结果。

示例：

| 任务 | 主技能 | 辅助技能 |
|---|---|---|
| 从开源项目沉淀技能 | open-source-to-skill-library | Prompt 01、Prompt 02、质量检查 |
| 下载视频并转格式 | video-download-skill | format-conversion-factory |
| 把网页工具变成本地应用 | web-project-to-desktop-app | format-conversion-factory、质量检查 |
| 建立新技能库工作流 | open-source-to-skill-library | 技能路由规则、多技能工作流模板 |

首页不再展示大块推荐工作流，但这些判断规则必须保存在注册表和路由规则中，并由我在执行任务时默认使用。

## 15. 发布和更新标准

每次新增或修改技能后，必须按以下流程处理：

1. 更新独立 `SKILL.md`。
2. 如有新 Prompt，更新独立 Prompt 文件。
3. 更新技能注册表 Markdown。
4. 更新 `skill-registry.html` 的卡片、弹窗、分类和统计。
5. 同步 GitHub 星数和最近更新日期。
6. 运行发布脚本。
7. 提交并推送 GitHub Pages 仓库。
8. 验证线上页面是否更新。

GitHub Pages 有缓存时，可用版本参数验证：

```text
https://macabyavaha7-sys.github.io/L-One-skills/?v=<commit>
```

## 16. 未来默认执行行为

以后只要用户在这个对话中提供 skill、prompt、AI 技巧、开源项目或可复用工作流，我默认执行：

1. 判断它是否应成为独立技能、独立 Prompt 或注册表备注。
2. 如果是技能，创建或更新独立 `SKILL.md`。
3. 如果是 Prompt，创建或更新独立 Prompt 文件。
4. 更新技能注册表、路由规则和可视化网页。
5. 补充 GitHub 来源、星数、最近更新日期和授权提醒。
6. 能找到可信推荐来源时补充 Top 5 引用；找不到时不编造。
7. 发布到 GitHub Pages，让网页保持最新。

这份标准就是后续“技能库管”的默认记忆和执行准则。
